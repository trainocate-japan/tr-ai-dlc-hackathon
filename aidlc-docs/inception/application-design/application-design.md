# ハマリーニング — アプリケーション設計（統合ビュー）

## アーキテクチャ概要

イベント駆動型サーバーレスアーキテクチャ。ユーザー向けAPIは同期処理、データ取り込み・AI生成は非同期イベント駆動で処理する。

         v                                        v
+------------------+                    +-------------------+
| CloudFront       |                    | API Gateway       |

+------------------+    +---------++---------++---------++---------++---------+
                        | Feed    || Course  ||Collect- || Quiz    ||Discovery|
                        | Lambda  || Lambda  ||ion Lamb || Lambda  || Lambda  |
                        +---------++---------++---------++---------++---------+
                              |         |         |         |         |
                              v         v         v         v         v
+-------------------------------------------------------------------+
|                         DynamoDB Tables                             |

**注**: API GatewayはEdge Optimized型のため、CloudFrontを前段に置かない。
API GW自体がCloudFrontエッジロケーションを活用してグローバル配信する。

## 技術スタック決定事項

| 決定事項 | 選択 | 理由 |
|----------|------|------|
| フロントエンド構造 | Next.js App Router | Server Components活用、最新のReactパターン |
| Lambda分割 | リソース×メソッド単位 | 細粒度で独立デプロイ可能、責務明確 |
| バッチトリガー | S3 → EventBridge → SQS → Lambda | イベント駆動、バッファリング、リトライ対応 |
| DynamoDB設計 | マルチテーブル | エンティティ分離で明確、再帰処理回避 |
| フィードアルゴリズム | サーバーサイド（Lambda） | ユーザー固有の閲覧履歴を考慮した選定が可能 |
| Bedrock呼び出し | DDB Streams → EventBridge → SQS → Lambda | 非同期ファンアウト、テーブル分離で再帰回避、Token Limit Exception回避 |
| API Gateway | Edge Optimized型 | CloudFront前段不要、エッジロケーション活用 |
| クイズ形式 | ○×（マルバツ）2択 | シンプルで回答しやすい、フィード体験を中断しない |
| ロギング/トレーシング | ADOT Lambda Layer + X-Ray | 分散トレーシング、構造化ログ、リクエスト全体の可視化 |
| Streams条件 | ランキング更新時はイベント発行しない | 不要なBedrock再生成を防止 |

---

## 主要設計原則

1. **イベント駆動**: データ変更が自動的に下流処理をトリガー
2. **テーブル分離**: 再帰処理を防止（courses → course-summaries/quizzes は別テーブル）
3. **非同期処理**: Bedrock呼び出しはSQS経由で非同期実行（タイムアウト回避、Token Limit Exception回避）
4. **細粒度Lambda**: リソース×メソッド単位で独立性確保
5. **DLQ**: 全SQSキューにDead Letter Queueを設定しエラー退避
6. **認証一元化**: API Gateway Cognito Authorizerで全エンドポイント保護

## 設計判断記録（ADR）

### ADR-001: 発見マップのデータストアにNeptuneを採用しない

**ステータス**: 決定済み  
**日付**: 2026-05-07  

**コンテキスト**: 発見マップ機能ではコース同士の関係性（同カテゴリ）をグラフ状にビジュアライズする。グラフデータベース（Amazon Neptune）の導入を検討した。

**決定**: Neptuneを採用せず、DynamoDBからLambda内でグラフ構造を都度構築する。

**理由**:
- MVP段階のデータ規模（1600コース × 100ユーザー）ではLambda内計算で十分なパフォーマンス
- Neptune最小コストが月$73〜$130に対し、DynamoDBオンデマンドは月$5〜$15（約10〜20倍のコスト差）
- Neptune Serverlessでも最低1NCU($87/月)の固定費が発生
- VPC内配置必須によるLambdaコールドスタート増加の懸念
- Gremlin/openCypherの学習コスト

**将来の再検討条件**:
- ユーザー数が1万人を超えた場合
- ソーシャル機能（ユーザー間の関係性）を追加する場合
- 協調フィルタリング型レコメンドを実装する場合
- グラフトラバーサルが3ホップ以上必要になった場合

## 月額コスト概算（100ユーザー、KPI達成時）

### 前提条件
| 項目 | 値 |
|------|-----|
| ユーザー数 | 100人 |
| 週次アクティブユーザー | 40人（リピート率40%） |
| 月間セッション数 | 40人 × 4週 × 3回/週 ≈ 480セッション/月 |
| セッションあたりAPIコール | 約40回（フィード取得×3 + 閲覧記録×30 + いいね/あとで見る×5 + その他） |
| 月間APIコール合計 | 約19,200回 |
| コースデータ | 1,600コース |
| Bedrock呼び出し | 初回一括 + 月数回の更新（約50回/月） |

|----------|--------|----------|

| **API Gateway** | 19,200リクエスト | $0.07 |

| **S3** | ストレージ: <1GB, リクエスト: 数百回 | $0.10 以下 |
| **CloudFront** | 転送量: ~5GB/月 | $0.50〜$1.00 |
| **Cognito** | 100 MAU（無料枠内: 50,000 MAU） | $0（無料枠） |
| **Amazon Bedrock** | 約50回/月 × 入力~2000トークン + 出力~500トークン | $1〜$3 |
| **SQS** | 数千メッセージ/月（無料枠: 100万/月） | $0（無料枠） |
| **EventBridge** | 数百イベント/月（無料枠: 1400万/月） | $0（無料枠） |

### 月額合計

| カテゴリ | 概算 |
|----------|------|
| コンピュート（Lambda） | ~$0.15 |
| API Gateway | ~$0.07 |
| データベース（DynamoDB） | ~$3〜$5 |
| ストレージ・配信（S3, CloudFront） | ~$1〜$2 |
| AI（Bedrock） | ~$1〜$3 |
| 監視（CloudWatch） | ~$0.50〜$1 |
| **合計** | **約 $6〜$12/月** |

### 備考
- 上記はAWS無料枠適用後の概算（アカウント作成12ヶ月以内ならさらに安い）
