# ハマリーニング — アプリケーション設計（統合ビュー）

## アーキテクチャ概要

イベント駆動型サーバーレスアーキテクチャ。ユーザー向けAPIは同期処理、データ取り込み・AI生成は非同期イベント駆動で処理する。

## 技術スタック決定事項

| 決定事項 | 選択 | 理由 |
|----------|------|------|
| フロントエンド構造 | Next.js App Router | Server Components活用、最新のReactパターン |
| Lambda分割 | リソース×メソッド単位 | 細粒度で独立デプロイ可能、責務明確 |
| バッチトリガー | S3 → EventBridge → SQS → Lambda | イベント駆動、バッファリング、リトライ対応 |
| DynamoDB設計 | マルチテーブル | エンティティ分離で明確、再帰処理回避 |
| フィードアルゴリズム | サーバーサイド（Lambda） | ユーザー固有の閲覧履歴を考慮した選定が可能 |
| Bedrock呼び出し | DDB Streams → EventBridge → SQS → Lambda | 非同期ファンアウト、テーブル分離で再帰回避 |

---

## 主要設計原則

1. **イベント駆動**: データ変更が自動的に下流処理をトリガー
2. **テーブル分離**: 再帰処理を防止（courses → course-summaries/quizzes は別テーブル）
3. **非同期処理**: Bedrock呼び出しはSQS経由で非同期実行（タイムアウト回避）
4. **細粒度Lambda**: リソース×メソッド単位で独立性確保
5. **DLQ**: 全SQSキューにDead Letter Queueを設定しエラー退避
6. **認証一元化**: API Gateway Cognito Authorizerで全エンドポイント保護
