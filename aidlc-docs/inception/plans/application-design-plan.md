### Question 2: バックエンドAPIの構成
Lambda関数の分割粒度はどのようにしますか？

A) 機能単位（フィードAPI、コレクションAPI、認証API、クイズAPI、管理APIなど機能ごとに分割）
B) リソース単位（courses, users, collections, quizzes などリソースごとに1つのLambda）
C) モノリシックLambda（1つのLambdaで全APIを処理、内部でルーティング）
D) おまかせ（最適なものを提案してほしい）
X) その他（[Answer]: タグの後に詳細を記述してください）

[Answer]: Bでさらにメソッド単位

### Question 3: バッチ処理のトリガー方式
S3にファイルが置かれた時のバッチ処理はどのように起動しますか？

A) S3イベント通知 → Lambda自動起動（ファイル配置で即時処理）
B) EventBridge Scheduler（定期実行、例: 毎日1回S3をチェック）
C) 手動トリガー（管理者がコンソールやCLIで実行）
D) A + B のハイブリッド（ファイル配置で即時 + 定期的な整合性チェック）
X) その他（[Answer]: タグの後に詳細を記述してください）

[Answer]: Aでイベント通知をEventBridgeに向けて、ターゲットアクションをSQS、キューをトリガーにLambda実行

### Question 4: DynamoDBのテーブル設計方針
テーブル設計のアプローチはどうしますか？

A) シングルテーブルデザイン（1テーブルに全エンティティ、GSIで多様なアクセスパターン対応）
B) マルチテーブルデザイン（エンティティごとにテーブル分離: courses, users, collections, quizzes等）
C) おまかせ（アクセスパターンに基づいて最適な設計を提案してほしい）
X) その他（[Answer]: タグの後に詳細を記述してください）

[Answer]: B

### Question 5: フィードのランダム化・多様性アルゴリズムの実行場所
フィードに表示するコースの選定ロジックはどこで実行しますか？

A) バックエンドLambda（APIリクエスト時にサーバーサイドで選定）
B) フロントエンド（全コースリストを取得し、クライアントサイドでシャッフル）
C) 事前計算（バッチでフィード順序を事前生成し、DynamoDBに保存）
D) おまかせ（最適なものを提案してほしい）
X) その他（[Answer]: タグの後に詳細を記述してください）

[Answer]: A

### Question 6: Bedrock呼び出しの方式
Bedrockによるコンテンツ生成（概要サマリー + クイズ）の呼び出し方式は？

A) コースデータ登録バッチと同一Lambda内で同期的に実行
B) コースデータ登録後にStep Functionsで非同期ワークフロー実行
C) 別のLambdaをSQS経由で非同期呼び出し（コース登録 → SQS → Bedrock生成Lambda）
D) おまかせ（最適なものを提案してほしい）
X) その他（[Answer]: タグの後に詳細を記述してください）

[Answer]: C コースデータ登録バッチLambdaでDynamoDB更新→DyanamoDBストリーム→Lambda→EventBridge→2つのSQS(クイズと概要サマリー)→それぞれのLambda関数で作成、対象となるテーブルは最初のコースデータ登録テーブルとは別で、概要サマリーカード用のテーブルとクイズ用のテーブルとして再帰処理を避ける。
